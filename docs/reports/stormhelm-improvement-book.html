Stormhelm Improvement Book

A comprehensive repo-wide review of what could be improved, clarified, expanded, stabilized, or made more complete

Audit date: 2026-04-20

Audit root: C:\Stormhelm

Method: Review of docs, configs, backend and UI source, tests, release files, generated directories, and current runtime wiring behavior.

Executive judgment: Stormhelm's biggest current opportunities are not lack of ambition. They are alignment, completion, and hardening: docs lag code, some advanced capabilities are only partially wired, the repo carries heavy generated clutter, and the UI test layer is not yet stable enough to certify the shell with confidence.

2. Product And Documentation Alignment

The repo's written story is behind the repo's actual implementation story. That is a quality issue, not just a marketing issue, because it makes it harder to reason about what is intentional, supported, or experimental.

What is misaligned today

  • The README reads like an early-phase foundation description, while the codebase already contains substantial workspace, diagnostics, workflow, power, and adaptive-UI machinery.
  • Specs and plans show deeper intent than the top-level public description does.
  • Some UI copy and module naming still sound future-facing even where partial implementation already exists.
  • The checked-in default config does not make the full effective tool surface obvious because several tool flags rely on model defaults rather than explicit config declaration.

Improvements to make

  • Create a single canonical capability matrix that marks each feature as implemented, enabled by default, experimental, or staged.
  • Rewrite the README around the actual Ghost-plus-Deck product, not just the earliest phase language.
  • Add a short architecture map that explains backend core, UI bridge, workspace memory, and deterministic tools in one place.
  • Make the module list explicit so "Helm", "Watch", "Systems", "Signals", "Files", and "Logbook" are understandable without reading the code.

3. Feature Exposure And Capability Wiring

Stormhelm has a non-trivial amount of code that is more advanced than the currently exposed runtime. That is exciting, but it also creates ambiguity about what really works end to end.

Capability stateObserved examplesImprovement opportunity
Implemented and reachableWorkspace memory, network diagnosis, desktop search, workflow execution, file operations, routines, trusted hooks, power diagnosis, provider-backed orchestrationDocument clearly and make these flagship features more visible in the UI and README.
Implemented but not default-enabledcontext_action, control_capabilities, window_control, window_statusEither enable and polish them, or explicitly mark them as staged.
Implemented in source but not registeredSystemControlTool exists with a declared tool name of system_control, but is not added to the builtin registry.Treat this as a concrete bug and close the gap between planner expectations and actual tool availability.
Intentionally disabledshell_command is a stub and disabled by policyKeep it disabled if that is the desired safety posture, but surface the limitation cleanly so operators do not infer non-existent shell execution powers.
Present as a UI placeholder or staged surfaceVisual-context panel model, some browser-and-research framing, some system-control ideasTurn placeholders into clear roadmap markers or ship the minimum viable implementation.

Root-cause detail behind the mismatch

  • ToolEnablementConfig explicitly models many advanced tools, but not every tool class present in the registry-facing source.
  • The builtin registry setup imports and registers many tool classes, but not SystemControlTool.
  • The planner already knows how to ask for system_control, window_control, and context_action, which means routing logic is ahead of final exposure discipline.
  • This is exactly the kind of drift that a centralized capability contract or manifest would prevent.

4. Reliability, Testing, And UI Stability

The codebase has a meaningful test suite, but the shell is not yet in a state where the UI layer can be trusted as a stable release gate.

Observed issues

  • A full pytest run in the project virtualenv reached the QML test layer and then hit a fatal Windows exception.
  • A reduced run excluding one unstable test still produced another crash and at least one explicit failure before teardown instability interrupted the session.
  • The instability appears tied to offscreen Qt or QML load and teardown behavior, not to simple unit-level logic.
  • The most UI-critical part of the product is therefore one of the least certifiable parts of the suite right now.

Recommended reliability work

  • Separate pure model-shaping tests from QML lifecycle tests so failures are more diagnosable.
  • Use stricter setup and teardown helpers for engine, root objects, and event pumping.
  • Add a focused shell smoke suite that validates boot, render, layout presets, and bridge bindings without fragile overreach.
  • Run packaged-app smoke tests as part of a release verification lane, not just source-mode tests.

5. Architecture And Maintainability

Stormhelm's architecture is good enough to ship from, but several files have become large enough that future velocity will slow unless they are split.

Large fileWhy it is largeRefactor opportunity
ui/bridge.py, 3,809 linesIt handles mode switching, message reconciliation, workspace modeling, deck layouts, system summaries, panel building, and adaptive ghost state.Split into bridge state, deck layout modeling, workspace surface builders, and transcript or signal shaping modules.
core/system/probe.py, 2,843 linesIt combines machine, power, resource, network, app, window, system, location, weather, and repair behaviors.Break into domain probes such as power, network, app control, location, and system-actuation services.
core/orchestrator/planner.py, 1,837 linesIt covers many families of natural-language routes in one place.Split intent families into weather, workspace, control, diagnostics, and workflow planners under a shared router.
core/workspace/service.py, 1,174 linesIt holds assembly, restore, save, archive, fuzzy matching, and next-step logic together.Separate persistence, inference, restore resolution, and summary derivation.
core/tools/builtins/system_state.py, 1,074 linesMany tool wrappers live together.Split by domain so machine, power, network, location, and control tools are easier to reason about.

Higher-level structural improvements

  • Define explicit domain boundaries between probe, planner, orchestration, and UI shaping.
  • Move toward more composable tool families with smaller files and stronger type boundaries.
  • Consider a declared capability registry model so the planner, config, builtin registry, and docs cannot drift apart.

6. User Experience, Accessibility, And Interaction Design

Stormhelm's visual ambition is a strength, but it also raises the bar for finish quality.

UX improvements worth making

  • Onboarding: explain Ghost vs Deck clearly on first run so operators know when to stay light and when to go deep.
  • Capability discoverability: expose the strongest features such as workspace memory, search, repair chains, and routines through guided prompts or command examples.
  • State transparency: make it clearer when a result came from local deterministic tools, remembered workspace state, or cloud reasoning.
  • Staged-feature honesty: if a panel is placeholder-level, say so directly in the interface instead of implying completion.
  • Error recovery: give clearer next-action buttons when a capability is disabled, unavailable on the current machine, or partially supported.

Accessibility improvements worth making

  • Keyboard navigation audit across Ghost, Deck, and panel chrome
  • Reduced-motion and lower-effects mode for heavy atmospheric rendering
  • Contrast validation for adaptive Ghost states on difficult backgrounds
  • Better screen-reader semantics where possible in the PySide and QML surface
  • Text-scaling sanity checks in the deck and transcript surfaces

7. Safety, Privacy, And Trust

Stormhelm already deals with sensitive local context such as clipboard content, selection data, location, network history, open apps, and remembered workspaces. That means its privacy and trust story should be made more explicit.

Trust opportunities

  • Explain what is stored locally and for how long, especially for workspace memory, tool history, location preferences, and telemetry trends.
  • Add clearer consent and fallback behavior for precise-location requests.
  • Show operators when a trusted hook is custom and what it will execute before running it.
  • Decide whether shell-style actuation will remain permanently stubbed or eventually become a controlled feature.
  • Harden the trusted-hook story further if this becomes user-facing, including better provenance and inspection.

8. Repository Hygiene, Packaging, And Release Operations

The current working tree contains a lot of generated material. Some of that is useful for local verification, but it also obscures the actual product surface.

Observed hygiene concerns

  • .tmp contains thousands of temp files and directories.
  • build contains many verification and packaging leftovers, including directories that hit permission issues during recursive traversal.
  • runtimeprobe contains a test database and journal.
  • .artifacts contains many screenshot and shader-check outputs that should be clearly treated as artifacts, not product source.
  • The presence of release binaries inside the repo tree may be intentional, but it blurs source vs output.

Release and packaging improvements

  • Codify what is expected to live in release/ vs what should be generated externally.
  • Add scripted cleanup for temp and verification leftovers.
  • Document the intended packaging flow from source to portable build to installer build.
  • Keep a clean separation between committed source assets and ephemeral build verification output.

9. Strategic Feature Additions

Stormhelm already has a strong skeleton. The next new features should deepen that skeleton rather than scatter attention.

High-leverage additions

  • Finish the visual-context story: turn the placeholder panel into a real image, screen-region, or desktop-scene context feature.
  • Complete the control surface: decide whether window control, system control, and context actions are shipping features, then wire them end to end.
  • Strengthen workspace intelligence: add richer task extraction, workspace timelines, and explicit resume summaries.
  • Deepen systems dashboards: use the telemetry and network history that already exist to power trend panels and predictive warnings.
  • Improve operator automation: routines and trusted hooks already exist; scheduling, templating, and safer review flows would compound their value.

Good candidate stretch features because the architecture already leans that way

  • more explicit mission boards or task boards inside Deck
  • richer browser and research surfaces tied to workspace context
  • historical power and network trend analysis
  • stronger note-to-workspace and note-to-thread linking
  • a clearer operator command language and surfaced routine catalog
