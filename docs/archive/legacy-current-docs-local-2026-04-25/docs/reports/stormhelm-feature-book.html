2. What Stormhelm Is, At A Product Level

Stormhelm is a desktop command assistant built around the idea that an operator should be able to get quick bearings from a light overlay surface, or open a richer command-and-context workspace when deeper interaction is needed. The product identity that emerges from the codebase is maritime and operational: helm, chartroom, watch, signals, systems, and logbook language all reinforce that Stormhelm is meant to feel like a standing command companion rather than a generic chat pane.

Core product thesis

  • Local-first runtime: the app boots with local tooling, local storage, local logs, and deterministic actions even when cloud reasoning is disabled.
  • Two-surface interaction model: Ghost mode gives fast overlay interaction; Deck mode hosts richer modules, previews, browser/file surfaces, workspace memory, and operational dashboards.
  • Operational assistant posture: Stormhelm does not just answer questions; it diagnoses, opens, restores, focuses, repairs, searches, archives, tags, saves, and assembles working context.
  • Deterministic first, model second: the planner tries to route obvious requests to structured tools before escalating to the reasoning model.
  • Memory as working context: the app treats workspaces, notes, opened files, browser targets, and task posture as persistent context rather than disposable transcript trivia.

Primary user-facing surfaces

SurfacePurposeWhat the code shows today
GhostMinimal ambient overlayDisplays prompt capture, quick bearings, context cards, recent message echoes, and adaptive placement and contrast based on the live desktop behind it.
DeckIntegrated command workspaceHosts modular panels, file/browser viewers, workspace canvas, systems and watch modules, layout presets, hidden-panel rail, and persistent panel geometry.
Core + strip presentationAmbient state communicationAnimated voice core, signal strip, corner readouts, atmospheric field layers, and transcript-linked status language.

3. Runtime And Architecture

The application is split into a backend core service and a PySide6 desktop UI. It uses a local API boundary rather than shoving everything into one process. The code is not huge at enterprise scale, but it is clearly layered and deliberate.

LayerMain filesRole
App launch and loggingapp/launcher.py, app/logging.pyBuilds command lines, chooses packaged vs source execution, and establishes process-specific logs.
Config loadingconfig/loader.py, config/models.pyMerges bundled defaults, optional overrides, .env, and user runtime config into structured application state.
Runtime path discoveryshared/runtime.py, shared/paths.pyDistinguishes source mode vs frozen mode, install root vs resource root, and default data paths.
Core APIcore/api/app.py, core/api/schemas.pyExposes backend endpoints the UI can poll and invoke.
Dependency containercore/container.pyWires database, probe, events, jobs, tools, provider, network monitor, and orchestration.
Desktop UIui/app.py, ui/main_window.py, ui/bridge.py, assets/qml/Main.qmlCreates the desktop app, loads QML, exposes bridge state and actions, and orchestrates the visual shell.

Startup is straightforward in concept but mature in implementation: runtime discovery establishes mode, config loader resolves bundled plus local settings, the core container initializes services and persistence, and the UI polls structured snapshots rather than scraping ad hoc process state.

4. Conversation, Orchestration, And Provider Behavior

The assistant stack is not just a chat loop. It is a structured message handler with context capture, deterministic tool routing, judgment hooks, optional model-backed reasoning, and persisted conversation state.

PartRoleObserved behavior
Intent routerLightweight classificationHelps separate requests that should be handled deterministically from those that need broader reasoning.
Deterministic plannerStructured tool planningMaps natural language onto weather, location, power, network, app control, workspaces, repairs, workflows, file operations, and more.
Assistant orchestratorMain request handlerCaptures context, records turns, invokes planner and tool runs, stores state, and optionally asks the provider for a reasoned answer.
Judgment servicePost-action advisoriesSuggests sensible next actions such as relaunching after a force quit or running a check after a repair.

OpenAI integration model

  • Stormhelm supports the OpenAI Responses API through core/providers/openai_responses.py.
  • The default configured planner model is gpt-5.4-nano.
  • The default configured reasoning model is gpt-5.4.
  • Previous response IDs are persisted so multi-turn reasoning can continue coherently.
  • Selection, clipboard, workspace context, and recent tool results can be injected into provider calls as structured grounding.
  • Cloud reasoning is optional; the local deterministic product remains meaningful even when provider access is disabled.

Response shaping

The orchestrator does not only store one big assistant string. It shapes output into visible tiers such as bearing title, concise response, micro-suggestion, and action metadata so Ghost and Deck can display operational answers compactly.

5. Workspace Memory And Active Context

Workspace memory is one of the strongest features in the codebase. Stormhelm has a real concept of a workspace, not just a filename list.

Workspace features implemented

  • Assemble workspace: infer a workspace from a query using indexed files and associated notes.
  • Restore workspace: bring back a recent or fuzzy-matched workspace by name.
  • Save workspace: persist the current posture and opened-item state.
  • Clear workspace: clear active posture without deleting historical records.
  • Archive workspace: mark older workspaces as archived.
  • Rename and tag: evolve workspace naming and categorization over time.
  • List workspaces: enumerate saved and archived work contexts.
  • Where left off: summarize saved state for a workspace.
  • Next steps: derive likely next actions from saved items and prior posture.
  • Alias learning: successful restores can teach the system alternative names for future retrieval.

What a workspace captures

  • Opened local files and their viewer metadata
  • Opened browser destinations
  • Focused workspace sections and destination modules
  • Note linkage
  • Likely next bearing and pending-step summaries
  • Saved deck layout state associated with that workspace

Active context understanding

The active context service classifies clipboard and selection data into useful categories such as URL, file paths, code, and plain text. The planner then uses that context for operations like opening what was copied, extracting tasks from a selection, or restoring context from active material.

6. System, Power, Network, Location, And Weather Capabilities

Stormhelm has a meaningful operational systems layer. It does not just call a generic system-info function; it maintains domain-specific tools and summaries for machine identity, power, resources, storage, network health, geographic bearings, and weather.

Machine and resources

  • Machine status: machine name, OS, platform, timezone, and local clock.
  • Resource status: CPU, memory, storage, thermal, and helper-enriched metrics when available.
  • Storage status: drive overview and free-space reporting.
  • Recent files: local file scan across allowed directories with explicit noise-folder skips.

Power and telemetry

  • Power status: battery percent, AC line state, time-to-empty, and time-to-full when available.
  • Power projection: estimates for time to a threshold percent, time to empty, live draw, and drain rate.
  • Operational diagnosis: dedicated power, resource, and storage diagnosis tool wrappers turn raw state into concise operator-facing explanations.
  • Hardware telemetry helper: separate helper path with idle, active, and burst cache tiers.
  • Telemetry overlays: helper data can enrich power and resource status outputs.
  • Trend support: helper side stores history for rolling estimates and recent points.

Network awareness

  • Background network monitor with persisted history
  • Gateway, external path, DNS, and Cloudflare quality comparisons
  • Diagnostic burst mode for higher-intensity samples
  • Event detection for disconnects, reconnects, roam-linked outages, gateway timeouts, packet-loss spikes, and DNS bursts
  • Attribution that distinguishes likely local-link issues, upstream instability, DNS failure, weak signal, or stable conditions
  • Formatting layers for concise bearings and richer Deck systems views

Location and weather

  • Saved home location and named saved locations
  • Best-location resolution that prefers live device bearings when available
  • Approximate fallback behavior and IP-based lookup path
  • ZIP-code and named-place lookup support
  • Current and near-term weather routing, including today, tomorrow, tonight, and weekend horizons
  • Weather source integration that produces answer text plus destination links

Location-resolution stack observed in code

  • live device location through Windows geolocation APIs when available
  • saved home and named saved locations in preferences
  • approximate device fallback
  • IP-based estimation via external lookup
  • ZIP lookup via a ZIP-code service
  • named-place geocoding via a general geocoder

Stormhelm does real things. The action surface is broader than the README implies.

Search and open behavior

  • Deck vs external routing: URLs and files can be opened internally or externally depending on operator request and current posture.

  • File viewer typing: text, markdown, image, and PDF viewing have dedicated handling.

8. User Interface And Adaptive Visual Shell

Stormhelm has a strong amount of visual implementation already. This is not a placeholder web panel with a few cards; it is a bespoke PySide6 and QML desktop shell with custom shaders, layered atmospherics, and bridge-driven state models.

Main QML composition

  • Main.qml builds a fullscreen frameless application window.
  • It layers StormBackground, GhostShell, CommandDeckShell, central voice and signal elements, and foreground mist.
  • It binds Ghost adaptive style and placement outputs into visible shell positioning and contrast behavior.

Key QML components currently present

  • Shell and navigation: GhostShell, CommandDeckShell, WorkspaceRail, CommandRail, CommandSpine
  • Panels and surfaces: DeckPanel, DeckPanelWorkspace, WorkspaceCanvas, ModulePanel, FileViewerSurface, BrowserSurface, NetworkHealthSurface
  • Ambient and visual layers: SeaFogField, StormBackground, glass and mist layers, and shader-backed fragment passes
  • Communication surfaces: PromptComposer, SignalStrip, TranscriptTimeline, CornerReadout, ContextCard
  • Interaction hubs: VoiceCore, layout presets, panel launcher, hidden panel rail, and opened-items strip

UI bridge behavior

The bridge acts like a view-model and state-translation layer for almost everything the UI displays. It tracks mode, messages, jobs, events, notes, context cards, opened items, workspace focus, layout state, adaptive ghost posture, and per-workspace panel geometry.

Deck layout system

  • Spatial deck modeled as 12 columns by 8 rows
  • Reserved anchor-clearance notch near the center-top zone so the Ghost and core cluster do not get visually collided with
  • Named layout presets such as Command, Workspace, Systems, and Research
  • Panel hide/show, collapse, pin, drag, resize, restore, reset, and saved layout snapshotting
  • Per-workspace layout persistence in a JSON store under the runtime data directory

Adaptive Ghost implementation

  • Background sampler grabs the live desktop behind candidate Ghost placements.
  • Readability scorer evaluates brightness, variance, contrast, edge density, and motion.
  • Placement controller chooses among center, left, right, and corner-adjacent candidate positions using thresholds, dwell time, and cooldown logic.
  • Style model adapts glow, contrast, surface strength, and text emphasis depending on the background state.
  • Reevaluation happens on a timed cadence, with smoothing to avoid jarring reposition behavior.

12. Further-Capable Features That Exist In Code But Are Not Yet Central In Current Usage

This is the section you explicitly asked for: places where Stormhelm is more capable than its current everyday surface suggests.

Capabilities present in source but not highlighted by the docs

  • Planner/reasoner split: the app already separates cheap deterministic and nano-planner work from heavier reasoning-model work.
  • Long-tail operator power: routines, trusted hooks, cleanup recipes, duplicate detection, archive flows, and file-grouping logic are more advanced than the current README implies.
  • Workspace intelligence: alias learning, fuzzy restoration, likely-next-bearing derivation, and where-left-off summaries indicate a stronger memory model than is currently marketed.
  • Operational awareness layer: there is a real interpreted-signal pipeline that converts raw system, event, and job state into watch-friendly summaries.
  • Adaptive desktop readability: Ghost placement and styling are driven by live desktop sampling rather than static theming.

Capabilities implemented but not yet first-class in the live everyday surface

  • Window inspection and movement: window-status and window-control tool classes exist in source, but they are not enabled by default in the current merged config.
  • Control-capabilities introspection: there is a tool class for reporting supported control groups, but it is disabled by default.
  • Context-action tooling: code exists to inspect, open, extract tasks from, and restore from active context, but the tool is not default-enabled.
  • System control: a SystemControlTool class exists in source and the planner knows how to request system_control, but the builtin registry does not currently register it.
  • Visual context module: the deck surface anticipates a stronger visual-context panel than is currently implemented.

Capabilities implied by architectural direction

  • Stormhelm is positioned to become a stronger local command center because the bridge, watch, systems, workspace, and workflow layers are already separated cleanly enough to deepen.
  • The current provider architecture could support richer tool-backed reasoning with minimal conceptual changes.
  • The deck layout and module system could support more domain surfaces without re-architecting the shell.
  • The telemetry and history machinery suggest future trend visualizations, predictive warnings, and richer operational dashboards.

14. Bottom-Line Summary

Stormhelm today is a substantial Windows desktop assistant platform with a clear interaction philosophy and a surprising amount of working operational depth. Its strongest current characteristics are:

  • a distinctive Ghost plus Deck desktop shell
  • deterministic operational planning before model escalation
  • real workspace memory instead of shallow session memory
  • strong systems and network diagnostics foundations
  • useful search, repair, workflow, file-operation, and maintenance tooling
  • portable packaging and local runtime discipline
  • a serious test suite, even if some UI paths remain unstable

Stormhelm is not finished, and the improvement book covers the gaps in detail. But the codebase already contains enough real product behavior that it should be described as an emerging desktop operating companion, not as a toy chat shell.
